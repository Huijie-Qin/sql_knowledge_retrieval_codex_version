## 详细需求描述

### 核心目标

设计一个能够本地运行的**解析数据源的python工程脚本**，从而达到如下的效果： 对./案例 目录下的所有源文件进行遍历，针对每一个源文件，进行**逐一处理**，每处理完成一个源文件后，立即更新数据源文件和解析进度。


### 执行流程

#### 1.初始化解析环境

1. **创建必要的目录结构**
	- 确保数据源目录存在：`数据源/`
	- **动态创建业务域子目录**
		- 首先扫描 `案例/` 目录，识别所有实际存在业务域（包括.md 以及.sql文件所在的目录）
		- 对于已知业务域（广告，应用，音乐，公共，全域搜索，电商），直接创建对应子目录
		- 对于未列出的业务域（如：视频，社交，游戏等），根据源文件中数据源被使用的内容，推理后创建相应业务子目录

2. **初始化解析进度文件**
	- 如果`数据源/解析进度.md`不存在，则创建该文件
	- 初始化格式如下：

```markdown
# 数据源解析进度

## 待解析文件

### 案例文件（.sql 或者 .md）
- [ ] 案例/电商行业案例/[文件名1].md
- [ ] 案例/电商行业案例/[文件名2].md
...




## 已解析文件
- [ ] [文件路径]

## 数据源索引
- [数据源表名] ：[业务域]

## 解析记录
｜数据源名称 ｜解析时间 ｜操作类型 ｜更新点｜ 
｜----------｜----------｜----------｜----------｜  

```

### 2.逐个文件进行处理


#### 处理单个文件的流程

对每个待解析的源文件，执行以下步骤：

1. **读取文件内容**
	- 识别文件的类型（有.md, .sql）

2. **解析数据源信息**
	- 针对.md类型的文件：
		- 识别源文件内容涉及的业务场景，分析目标
		- 提取所有数据表（表名，数据库）
		- 提取关键字段以及用途
		- 记录SQL查询模式和使用方法
		- 提取数据质量相关信息（如果有的话）
		- 记录 源文件名 - 数据源 的关联关系
	- 针对.sql类型的文件
		- 从文件名推断业务场景（比如：analysis_ecommerce_表示电商分析，ads_persona_表示广告画像）
		- 解析SQL语句，提取所有表名（FROM，JOIN，INSERT INTO等字句中的表）
		- 识别关键字段（SELECT子句中的字段，特别是用于JOIN，WHERE，GROUP BY的字段）
		- 提取SQL查询模式（如：用户画像构建，漏斗分析，流量洞察，人群标签定义等）
		- 从SQL注释中提取业务说明（如果有的话）
		- 记录SQL文件作为源文件 - 数据源 的关联

3.更新或者创建数据源文件
	- 对每个识别出来的数据源（以表名为标识）：
		- **如果是新数据源**：在对于业务域目录下创建新的数据源文件，操作类型为 “新建数据源”
		- **如果是已经存在的数据源**：读取现有的同名数据源文件，合并新信息后更新此文件，操作类型为“更新数据源”。（注意：合并文件时如果需要调用LLM，则需要控制好避免上下文爆炸）
	- **更新点记录规则**（仅当操作类型为：“更新数据源”时）：
		- 对比新旧数据源文件内容，识别本次更新的具体内容：
		- 更新点描述应该明了，使用以下格式：
			- “新增SQL示例”：新增了SQL查询示例
			- “补充字段说明”： 补充了字段的用例说明
			- “更新数据质量信息”： 更新了数据量，覆盖情况或上报及时性等信息
			- “新增关联案例”： 新增了案例关联信息
			- “完善使用信息”：完善了使用方法和注意事项
			- “修正描述信息”：修正了数据源描述或字段说明
			- “合并重复信息”：合并了重复的SQL示例或字段信息
		- 多个更新点用分号分隔，如：“新增SQL示例；补充字段说明”
	- 数据源文件格式见 “输出文件格式”章节


4. 更新解析进度文件
	- 将当前文件从“待解析文件”移动到“已解析文件”
	- 更新“数据源索引”部分，记录新发现的数据源
	- **添加解析记录**，格式如下：
		- **数据源名称**：数据源表名
		- **解析时间**： 格式为yyyyMMdd-HHmmss
		- **操作类型**： “新建数据源” 或 “更新数据源”
		- **更新点**：仅当操作类型为 “更新数据源”时填写，记录本次更新的内容，尽量简洁（如： “新增SQL示例”，“补充字段说明” 等）

### 3.检查重复和遗漏

所有文件处理完成后，执行检察：

1.**读取已经梳理好的数据源文件**：遍历 `数据源/`目录下各个业务域子目录中的所有数据源文件
2.**识别重复**：
	- 检察是否有相同表名但不同文件名的数据源
	- 检察是否有内容高度相似的数据源
	- 如发现重复，则合并为一个数据源文件。 （注意：合并文件时如果需要调用LLM，则需要控制好prompt，避免上下文爆炸）
3.**识别遗漏**：
	- 对比案例中使用的所有数据表和已经梳理好的数据源列表
	- 识别案例中使用但未在数据源目录中出现的表
4.**生成检察报告**： 在`数据源/检察报告.md`中记录：
	- 发现的重复数据源以及处理结果
	- 发现的遗漏数据源列表
	- 数据源完整性评估
